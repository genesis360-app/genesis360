1. Visión general del producto

Propuesta de valor

Genesis360 es una plataforma SaaS de gestión para comercios físicos (retail, almacenes, distribuidoras, etc.). Su diferencial no es mostrar datos — es decirle al dueño qué hacer: qué producto reponer, qué margen está bajando, qué empleado cumpleaños tiene esta semana, qué caja tiene diferencia.

"El cerebro del negocio físico" — Insights automáticos, alertas proactivas y gestión integral en una sola plataforma.

Stack tecnológico

Frontend

  • React 18 + TypeScript
  • Vite (build tool)
  • Tailwind CSS + shadcn/ui
  • Zustand (estado global)
  • React Query (cache/fetching)
  • Recharts (gráficos)

Backend

  • Supabase (PostgreSQL + Auth)
  • Row Level Security (RLS)
  • Edge Functions (Deno)
  • Storage (archivos/imágenes)
  • Vercel (deploy frontend)
  • Mercado Pago (suscripciones)

Dominios

  • www.genesis360.pro — Landing page (marketing)
  • app.genesis360.pro — Aplicación (usuarios autenticados)

Arquitectura multi-tenant + multi-sucursal

Cada cliente es un tenant con datos completamente aislados. Todas las tablas tienen tenant_id con Row Level Security (RLS) en PostgreSQL. Desde v0.42.0, cada tenant puede tener múltiples sucursales, con filtrado por sucursal en inventario, ventas, caja, gastos y clientes.

3. Módulos actuales

📦 Inventario

Gestión de productos con múltiples líneas de stock (LPN — License Plate Number). Soporta productos serializados (número de serie único) y no serializados. Cada línea tiene ubicación, vencimiento, lote, cantidad reservada y estado.

  • Reglas de selección: FIFO / LIFO / FEFO / LEFO / Manual (por SKU o por negocio)
  • Prioridad de ubicación: sort por prioridad en picking
  • Aging profiles: cambio automático de estado según días transcurridos
  • Importación masiva por Excel (productos + series)
  • Scanner de código de barras (cámara, iOS/Android/Desktop)
  • Scan con IA: foto del producto → reconocimiento automático vía Claude Haiku + Open Food Facts

🔄 Movimientos de stock

Registro de ingresos y rebajes de inventario con trazabilidad completa.

  • Motivos predefinidos configurables + campo libre "Otro"
  • Vinculación a línea de inventario (LPN) y serie
  • Límite de movimientos por plan (con barra de progreso y bloqueo)
  • Banner de alerta al acercarse al límite

🛒 Ventas

Punto de venta con carrito, combos, descuentos y múltiples métodos de pago.

  • Estados: pendiente → reservada → despachada / cancelada
  • Carrito con descuento por ítem y descuento general
  • Combos: descuento por porcentaje, monto ARS o monto USD
  • Múltiples métodos de pago: Efectivo / Tarjeta / Transferencia / Mercado Pago
  • Trazabilidad LPN→venta (qué línea de inventario se usó)
  • Vista lista + galería de productos
  • Ticket de venta imprimible
  • Historial con filtros por estado, fecha y categoría
Regla clave: Reserva/despacho requieren cliente registrado (excepto despacho directo). El monto total debe estar cubierto al despachar.

💰 Caja

Gestión de caja física con sesiones de apertura/cierre, movimientos manuales y conciliación automática con ventas.

  • Múltiples cajas por sucursal (límite según plan)
  • Apertura con monto inicial · Cierre con conteo real
  • Ingresos manuales (con motivo) · Egresos manuales (con motivo)
  • Solo Efectivo afecta el saldo; otros medios son informativos
  • Ingreso automático al despachar venta con efectivo
  • Egreso automático al registrar gasto en efectivo
  • Widget de estado de caja en header (punto verde/rojo)
  • Multi-usuario: badge si sesión abierta por otro usuario
  • Shortcuts: Shift+I ingreso · Shift+O egreso

💸 Gastos

Registro de gastos del negocio con integración automática a caja cuando el medio es efectivo.

  • Categorías y proveedores configurables
  • Bloqueo si medio es efectivo y no hay caja abierta
  • Egreso automático en caja_movimientos (fire-and-forget)

👥 Clientes

Base de clientes del negocio con cuenta corriente.

  • Datos: nombre, DNI (único), teléfono, email, dirección
  • Importación masiva por Excel/CSV
  • Cuenta corriente: saldo de deuda sin interés ni vencimiento
  • Filtro por sucursal

🔔 Alertas

Sistema de alertas automáticas para gestión proactiva del negocio.

  • Stock mínimo: productos por debajo del umbral configurado
  • Stock cero: productos sin unidades disponibles
  • Productos sin categoría asignada
  • Ventas pendientes (sin fecha de vencimiento, requieren gestión manual)
  • Clientes con deuda pendiente (con link directo a ficha)
  • Diferencia de cierre de caja (notificación a OWNER/SUPERVISOR)

📊 Dashboard e Insights

Panel principal con métricas del negocio e insights automáticos generados por reglas de negocio.

Tab General

  • KPIs: ventas del período, ganancia neta, gastos, stock crítico
  • Proyección de cobertura (semáforo: rojo ≤7d / ámbar ≤14d / verde >14d)
  • Sugerencia de pedido para productos críticos

Tab Métricas

  • Top productos por ventas · Ranking por margen
  • Productos sin movimiento · Insights de margen vs. objetivo
  • Breakdown de movimientos por tipo/motivo
  • Valor de inventario por ubicación
  • Rango de fechas personalizable (7d / 30d / 90d / mes / custom)
  • Filtro por categoría

Tab Insights (IA de reglas)

  • Score de salud del negocio (Rotación / Rentabilidad / Reservas / Crecimiento / Datos)
  • 11 reglas automáticas: cobertura crítica, margen bajo, día flojo, cumpleaños del mes, stock sin movimiento, etc.

📋 Reportes

Exportación de datos a Excel y PDF. Disponible desde plan Básico.

  • Stock actual (con N° Lote, vencimiento, series)
  • Ventas del período (con breakdown por método de pago)

4. Reglas de negocio relevadas

Módulo: Caja

Límites y acceso

  • Roles con acceso: OWNER, SUPERVISOR, CAJERO. RRHH sin acceso.
  • Un cajero solo puede tener una caja abierta a la vez.
  • Cajas por sucursal: Free=1, Básico=2, Pro=3, Premium=5, Enterprise=10.

Apertura y cierre

  • Saldo sugerido al abrir = último cierre de esa caja específica (no la más reciente del sistema).
  • CAJERO solo puede cerrar su propia caja. OWNER/SUPERVISOR cierran cualquiera.
  • Cierre de caja ajena requiere contraseña maestra definida por el OWNER (pendiente implementar).
  • Al cerrar: se registra conteo real del cajero. Si hay diferencia → solo se registra + alerta a OWNER/SUPERVISOR. Sin bloqueo ni flujo de aprobación.

Ticket de cierre

  • Se genera comprobante/ticket de cierre imprimible (PDF).
  • Incluye: fecha/hora apertura y cierre, cajero, saldo inicial, ingresos, egresos, saldo sistema, conteo real, diferencia.
  • Puede reimprimirse desde el historial de sesiones (pendiente implementar).

Arqueo parcial

  • El cajero puede consultar si el efectivo físico cuadra con el sistema sin cerrar la sesión.
  • No queda registrado. Es una consulta momentánea (pendiente implementar).

Integración con ventas y gastos

  • Solo Efectivo afecta el saldo real de caja. Otros medios = registro informativo.
  • Sin caja abierta: no se puede despachar venta en efectivo ni registrar gasto en efectivo.
  • Venta despachada con efectivo → ingreso automático en caja.
  • Gasto en efectivo → egreso automático en caja.
  • Nómina en efectivo → egreso automático + verificación de saldo.

Módulo: Ventas

EstadoCliente requeridoPago requeridoEditable
Pendiente✅ ObligatorioNo✅ Sí
Reservada✅ ObligatorioParcial o total✅ Sí
Despachada❌ Opcional100% obligatorio❌ No

Pago parcial en reservas

  • Se puede registrar una reserva con pago parcial o total.
  • El pago entra a la caja inmediatamente con motivo "Pago de reserva de venta #X".
  • Al despachar, el sistema calcula y muestra automáticamente el saldo pendiente (total − ya pagado).

Registro de cliente inline

  • Si el cliente no existe, se puede registrar directamente desde la pantalla de venta.
  • Campos obligatorios: nombre completo, DNI (identificador único), teléfono.
  • Email: opcional.

Ventas pendientes

  • No vencen automáticamente. Aparecen en Alertas para gestión manual.

Módulo: Clientes — Cuenta corriente

  • Un cliente puede generar deuda (llevar mercadería con pago parcial o sin pagar).
  • La deuda no tiene vencimiento ni interés (por ahora).
  • Se salda desde la ficha del cliente.
  • Alertas automáticas de clientes con deuda + link directo a ficha.
  • Dashboard: total deuda a cobrar + % ventas pagadas vs. en deuda.

6. Arquitectura técnica

Multi-tenant

Todas las tablas tienen tenant_id UUID NOT NULL con RLS habilitado. El patrón de política es:

tenant_id IN (SELECT tenant_id FROM users WHERE id = auth.uid())

Las funciones helper is_admin() e is_rrhh() son SECURITY DEFINER para evitar recursión en RLS.

Multi-sucursal

Desde v0.42.0, 6 tablas operativas tienen sucursal_id UUID nullable: inventario_lineas, movimientos_stock, ventas, caja_sesiones, gastos, clientes. Sin sucursal seleccionada = vista global de todos los datos.

Seguridad

  • RLS en todas las tablas de datos de negocio
  • Triggers SECURITY DEFINER para operaciones que necesitan privilegios elevados
  • Edge Functions con JWT verificado (excepto webhooks públicos)
  • Variables de entorno en Vercel (frontend) y Supabase secrets (Edge Functions)
  • Repo público: sin credenciales en código
