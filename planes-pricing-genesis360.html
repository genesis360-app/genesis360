  • Netegia: comprobantes + artículos + sucursales + CUITs + proveedores + integraciones + usuarios (multi-dimensional; 4.000 artíc en el plan más barato).
  • Ninox: artículos + terminales POS + sucursales (extra $25k/suc); 2.000 artíc en el más barato.

  • ⇒ SKU/artículos SÍ es dimensión válida, pero a niveles de MILES, no cientos. Sucursales y usuarios son palancas comunes.

Posicionamiento de Genesis360

  • Gana: WMS real (LPN/lotes/series/FIFO-FEFO), POS + caja/bóveda, compras (OC+recep), envíos con reparto propio, multi-sucursal server-side. Nadie del segmento SMB junta todo (Netegia es el que más se acerca, y es 1,6-2,3x más caro).
  • Liviano: contabilidad pura (sin plan de cuentas/asientos) y sueldos con convenios. No es la pelea: competir contra "el comercio que quiere operar + facturar sin 3 sistemas", no contra el contador.

  • Fijo (recurrente): se suma a la factura mes a mes. Para bajarlo, el usuario debe desactivar recursos hasta la cantidad cubierta por el base (la app lo guía). SKU/sucursales/usuarios = solo fijo.
  • Temporal (one-time): pago en el momento, vence a 30 días del pago, independiente del ciclo mensual. Solo para movimientos.

Ejemplo: Pro con 10.000 SKU (base 8.000 + add-on 2.000×$10k) = $110.000/mes. Básico + 2ª sucursal (1×$15k) = $75.000/mes.

✅ Decisiones GO (2026-07-01):
  • Downgrade guiado (no borrado) — REGLA #0: al bajar de un add-on fijo, la app indica cuántos recursos debe DESACTIVAR el usuario para poder bajarlo; en SKU alerta "desactivar ≠ eliminar" (preservar trazabilidad). El usuario ejecuta, guiado.
  • Sucursal: precios finales 1×$15.000 / 3×$35.000 / 5×$55.000 (per-unit decreciente ✓).
  • Temporal solo para movimientos; SKU/sucursal/usuario = solo fijo.
  • Pro base SKU = 8.000 (subido de 6.000).
  • UX = configurador (plan base + selectores), no planilla.

3. Modelo de límites recomendado

Correr la monetización de "recursos artificiales" (SKU/movimientos) a VALOR real (módulos + usuarios + sucursales).

  • SKU y comprobantes → generosos o ilimitados (cuestan ~$0; sirven de marketing: "productos ilimitados", "facturación ilimitada").
  • Movimientos → límite suave (opaco para el cliente; hoy sin guard server-side).
  • Palancas fuertes de precio: módulos avanzados (WMS, RRHH, Compras, Envíos), cantidad de usuarios y sucursales, integraciones.

4. ¿Los límites aprietan la infra? (Supabase/Vercel/Resend)

No. A 0-100 clientes los límites son 100% palanca comercial, no restricción de infra.

  • SKU / movimientos / usuarios = filas Postgres + auth → baratísimo (100k productos ≈ decenas de MB).
  • Lo que escala el costo (en orden), recién con decenas-cientos de tenants activos: (1) Resend (free 3.000 emails/mes → ~50-100 tenants lo rompen → US$20/mes 50k); (2) Supabase storage/egress (imágenes de producto); (3) DB size (movimientos, años a este ritmo).
  • Riesgo real = límites demasiado BAJOS (los 100 SKU), no quedarse sin infra.

5. ¿Comprobantes como límite? Costo por comprobante = $0

AFIP/ARCA no cobra por CAE. Con AfipSDK el token es por-tenant (lo paga el cliente); con el WSFE propio (en construcción) → $0 marginal nuestro. Recomendación: NO limitar como tope punitivo → usar "facturación ilimitada" como diferenciador (Xubio capa 1.000/mes, Contabilium 2.500-10.000).

6. Costos propios (para el margen)

  • Storage: despreciable (~US$0,02/GB/mes; un Pro a tope ≈ US$0,06/mes). Facturas/remitos no se guardan (se regeneran).
  • Facturación AFIP: $0 de AFIP/ARCA. AfipSDK = token por-tenant (costo del cliente); WSFE propio = $0.
  • Comisión Mercado Pago: ~4,3% de cada cobro de suscripción. Cobro en ARS.
  • Hoy: solo Claude Code (US$23/mes) + dominio (~US$15/año); resto en free tier; 0 clientes en PROD.
